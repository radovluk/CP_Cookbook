# RCPSP with Time-Offs -- Benchmark Instances

This directory contains benchmark instances for the **Resource-Constrained Project Scheduling Problem with Time-Offs (RCPSP-TimeOffs)**. The problem extends classical RCPSP by introducing per-unit availability calendars -- each individual resource unit has periods of unavailability (time-offs) that must be respected when scheduling tasks.

## Instance Format (`.data` files)

Each `.data` file follows this structure (lines starting with `#` are comments):

```
# HEADER: <num_tasks> <num_types> <num_units>
8 1 2

# RESOURCE TYPES
# Format: <type_id> <num_units> <unit_id1> <unit_id2> ...
0 2 0 1

# RESOURCE UNITS (Calendars)
# Format: <unit_id> <num_steps> <t1> <v1> <t2> <v2> ...
0 4 0 100 35 0 40 100 70 0
1 4 0 100 58 0 62 100 70 0

# TASKS
# Format: <task_id> <duration> <num_requirements>
# Then for each requirement: <type_id> <quantity>
0 0 1
 0 0
1 5 1
 0 1

# PRECEDENCES
<count>
<pred_task_id> <succ_task_id>
```

**Calendar encoding:** Step-function values where `100` = available and `0` = unavailable. For example, `0 100 35 0 40 100 70 0` means: available from time 0 to 35, unavailable from 35 to 40, available from 40 to 70, then unavailable from 70 onward.

**Tasks:** Every instance includes a dummy start task (id 0, duration 0) and a dummy end task (last id, duration 0) with zero resource requirements. All work tasks are connected to both dummies via precedence constraints.

## Benchmark Categories

The benchmark suite contains **1,150 instances** across **15 categories**, organized into progressive difficulty tiers and special-purpose challenge sets.

### Difficulty Tiers (Tiers 1--10)

These tiers scale progressively in size and complexity. Tiers 1--5 (100 instances each) were generated by `generate_100_each.py`, and Tiers 6--10 (50 instances each) by `generate_large_benchmarks.py`.

| 1 | `tier1_trivial/` | 100 | 3--6 | 1 | 2--3 | 55--70 | Chain, Parallel | Uniform, Staggered |

| 6 | `tier6_large/` | 50 | 80--150 | 4--6 | 6--10 | 500--1,000 | Random DAG, Grid | Random, Staggered, Heavy Start/End |
| 7 | `tier7_industrial/` | 50 | 150--300 | 5--8 | 8--15 | 1,000--3,000 | Random DAG, Grid | Random, Staggered, Clustered |
| 8 | `tier8_mega/` | 50 | 300--600 | 6--10 | 10--20 | 2,000--5,000 | Random DAG, Grid | Random, Staggered |

**Scaling notes:**
- Horizon is computed as a function of the number of tasks (e.g., `horizon = 150 + num_tasks * 5` for Tier 4). Tiers 6--10 use explicitly sampled horizon ranges.
- Precedence density decreases as instance size grows (from 0.1--0.35 in small tiers to 0.01--0.04 in Tier 10) to keep graph construction tractable.
- Calendar breaks scale proportionally to the horizon in Tiers 6--10 via `horizon // break_interval`.
- Scarcity factor ranges from 0.8 (excess capacity) to 1.4 (tight), increasing with tier difficulty.

## How the Benchmark Generator Works

The generator source code is in `benchmarks/rcpsp-timeoffs/benchmark_generator/`. It consists of:

| File | Purpose |
|------|---------|
| `benchmark_generator.py` | Core `BenchmarkGenerator` class, `BenchmarkConfig` dataclass, and `save_instance()` |
| `generate_100_each.py` | Generates Tiers 1--5 + 3 special categories (800 instances) |
| `generate_large_benchmarks.py` | Generates Tiers 6--10 + 2 special categories (350 instances) |
| `solution_validator.py` | Validates solutions for all 6 problem variants |
| `usage_example.py` | Demonstrates generation, loading, statistics, and validation |

### Generation Pipeline

The `BenchmarkGenerator` class takes a `BenchmarkConfig` and produces an `Instance` through four steps:

1. **Resource structure** (`_generate_resources`): Creates `num_resource_types` types, each with a specified number of units. Units are assigned sequential IDs across all types.

2. **Task generation** (`_generate_tasks`): Creates work tasks with random durations in `[min_duration, max_duration]`. Each task's resource requirements are determined per type: with probability `requirement_probability`, the task requires `[min_requirements, max_requirements]` units of that type (scaled up by `scarcity_factor` if > 1). Every task is guaranteed at least one non-zero requirement.

3. **Precedence generation** (`_generate_precedences`): Builds a DAG from dummy start to dummy end using one of six topology strategies:
   - **Chain**: Linear sequence `start -> 1 -> 2 -> ... -> end`
   - **Parallel**: All tasks connected only to start/end (maximum parallelism)
   - **Tree**: Hierarchical branching from start, leaves connected to end
   - **Diamond**: Fan-out/fan-in groups creating convergence points
   - **Random DAG**: All tasks connected to start/end, plus random edges between task pairs with probability `precedence_density`
   - **Grid**: Tasks arranged in a 2D grid with right and down edges

4. **Calendar generation** (`_generate_calendars`): Generates per-unit availability step-functions using one of six patterns:
   - **Uniform**: Breaks at evenly spaced intervals
   - **Staggered**: Like Uniform, but offset per unit (so different units are unavailable at different times)
   - **Clustered**: All units have breaks at approximately the same times
   - **Random**: Breaks placed at random positions
   - **Heavy Start / Heavy End**: Breaks concentrated at the beginning or end of the horizon (exponential distribution)

   Break lengths are sampled around `avg_break_length` with variance, and adjusted by `calendar_fragmentation` (high fragmentation = more, shorter breaks). Calendars are normalized to remove overlaps and ensure proper step-function format.

### Reproducibility

All generation is seeded. Each category uses a distinct `base_seed` range (e.g., Tier 1 uses seeds 1000--1099, Tier 2 uses 2000--2099, etc.), ensuring instances are fully reproducible and independent across categories.

### Solution Validator

The `solution_validator.py` module validates solutions against 6 problem variants:

| # | Variant | Migration? | Delays? | Resource During Pause |
|---|---------|-----------|---------|----------------------|
| 1 | No Migration, No Delays | No | No | N/A |
| 2 | Migration, No Delays | Yes | No | N/A |
| 3 | No Migration, Delays, Blocked | No | Yes | Blocked (reserved) |
| 4 | Migration, Delays | Yes | Yes | N/A |
| 5 | Heterogeneous | Per-type | Per-type | Per-type |
| 6 | No Migration, Delays, Released | No | Yes | Released (freed) |

The validator checks: task completeness, correct work amounts, precedence satisfaction, resource conflicts (no double-booking), and variant-specific constraints (migration, continuity, calendar availability, blocked/released semantics).
